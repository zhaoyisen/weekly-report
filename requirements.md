## 2. 项目背景

用户在日常工作中会随手记录“完成了什么”“正在做什么”“计划做什么”。这些原始记录通常是口语化、碎片化、不适合直接放入周报的内容。

本工具希望提供一个 Windows 桌面应用，支持用户快速输入自然口语记录，并通过自定义 AI 大模型 API 将其优化为适合周报、月报或工作总结使用的规范任务条目。同时，系统需要按年、月、周、日管理工作记录，方便回溯、整理和导出。

## 3. 产品目标

1. 提供轻量、稳定、易用的 Windows 桌面记录工具。
2. 支持用户用口水话快速记录工作事项，降低记录成本。
3. 通过 AI 大模型将原始记录优化为正式、简洁、可用于周报的任务描述。
4. 支持按年、月、周、日查看、筛选、编辑和管理任务记录。
5. 支持用户自定义 AI API 地址、模型、密钥和参数，不绑定单一模型厂商。
6. 为后续增加周报自动生成、模板管理、导出等能力保留扩展空间。

## 4. 用户角色

| 角色 | 描述 | 主要诉求 |
| --- | --- | --- |
| 普通用户 | 需要记录个人工作内容并整理周报的员工、项目成员、管理人员 | 快速记录、自动润色、按周期整理、导出周报素材 |
| 管理型用户 | 需要整理团队或多个项目工作记录的用户 | 分类管理、周期统计、统一输出格式 |

V1.0 以单机个人使用为主，不设计多人协作和账号体系。

## 5. 业务范围

### 5.1 V1.0 范围内

1. Windows 桌面应用安装和本地运行。
2. 任务记录的新增、编辑、删除、查询、筛选。
3. 按年、月、周、日组织和查看任务。
4. 原始口语记录保存。
5. AI 优化后的周报任务条目保存。
6. 自定义 AI API 配置。
7. AI 优化失败时保留原始记录并允许重试。
8. 本地数据持久化。
9. 基础导出能力，至少支持复制周报文本或导出 Markdown。

### 5.2 V1.0 范围外

1. 多用户登录、权限、组织架构。
2. 云端同步。
3. 移动端 App。
4. 自动读取聊天软件、邮件、日历等外部数据。
5. 复杂项目管理功能，如甘特图、工时审批、绩效系统集成。
6. AI 模型训练或私有化部署。

## 6. 推荐技术方案

基于“Windows 桌面版”和“使用主流技术”的要求，推荐采用以下技术栈：

| 层级 | 技术选型 | 说明 |
| --- | --- | --- |
| 桌面框架 | Electron | Windows 桌面应用最主流方案之一，生态成熟，打包、自动更新、系统托盘等能力完善。 |
| 前端框架 | React + TypeScript | 主流、稳定、生态完善，适合构建复杂交互界面。 |
| 构建工具 | Vite | 启动快、配置轻，适合现代前端工程。 |
| UI 组件 | Ant Design 或 shadcn/ui | Ant Design 更适合后台和效率工具；shadcn/ui 更适合高度定制界面。V1.0 推荐 Ant Design。 |
| 本地数据库 | SQLite | 单机桌面应用常用方案，轻量、稳定、无需独立服务。 |
| ORM/数据访问 | Drizzle ORM 或 Prisma | 提高数据库结构维护和查询安全性。V1.0 可优先 Drizzle ORM。 |
| AI 调用 | 自定义 OpenAI-Compatible API Client | 支持用户填写 Base URL、API Key、Model，自适配兼容 OpenAI 格式的模型服务。 |
| 打包发布 | electron-builder | 主流 Electron 打包工具，支持 Windows 安装包。 |

## 7. 核心业务流程

### 7.1 快速记录流程

1. 用户打开应用。
2. 用户在快速输入框中输入口语化内容。
3. 用户选择记录类型：已完成、进行中、计划中、问题风险、临时想法。
4. 用户选择或默认使用当前日期。
5. 系统保存原始记录。
6. 系统调用 AI API 生成周报可用任务条目。
7. 用户确认、编辑或重新生成。
8. 系统保存最终任务条目。

### 7.2 周报整理流程

1. 用户进入“周视图”。
2. 系统展示当前周所有任务记录。
3. 用户按状态、项目、标签、日期筛选。
4. 用户选择需要进入周报的任务。
5. 系统按模板生成周报文本。
6. 用户复制或导出。

### 7.3 AI 配置流程

1. 用户进入“设置 - AI 配置”。
2. 用户填写 API 地址、API Key、模型名称和参数。
3. 用户点击“测试连接”。
4. 系统发送测试请求。
5. 测试成功后保存配置。
6. 测试失败时展示错误原因和排查建议。

## 9. AI 输出要求

### 9.1 单条任务优化输入

用户输入示例：

```text
今天把客户说的那个导出报错修了一下，顺便看了下日志，应该是字段为空导致的。
```

AI 输出示例：

```text
修复客户反馈的数据导出异常问题，定位为空字段导致的报错原因，并完成相关日志排查与处理。
```

### 9.2 输出原则

1. 不添加用户没有表达过的结果、数据、客户名称、系统名称。
2. 保持一句话或短句表达，适合作为周报条目。
3. 优先体现动作、对象、结果。
4. 避免过度口语化表达。
5. 避免夸大，如“显著提升”“全面优化”等无依据描述。
6. 对计划类事项使用“计划”“准备”“拟”等表达。
7. 对进行中事项使用“推进”“跟进”“处理中”等表达。
8. 对已完成事项使用“完成”“修复”“整理”“交付”等表达。

### 9.3 推荐 Prompt 规则

系统应内置可配置 Prompt，基础规则如下：

```text
你是一个工作周报整理助手。请将用户输入的口语化工作记录，改写为适合周报使用的一条正式工作任务描述。

要求：
1. 保留原始含义，不编造事实。
2. 表达简洁、正式、结果导向。
3. 输出一条中文任务描述，不要解释。
4. 根据用户标记的状态调整措辞：已完成、进行中、计划中。
5. 如果原文信息不足，保持概括，不要补充不存在的细节。
```

## 11. 页面与交互需求

### 11.1 主界面

1. 左侧为导航：今日、本周、本月、本年、全部记录、项目、设置。
2. 顶部提供快速输入框。
3. 中间展示当前周期任务列表。

### 11.2 快速输入区

1. 支持多行输入。
2. 支持选择状态和日期。
3. 支持一键保存并 AI 优化。
4. 支持仅保存原文，稍后再优化。

### 11.3 设置页

1. AI API 配置。
2. 周报模板配置。
3. 默认偏好设置。
4. 数据备份与恢复入口。

## 13. 验收标准

1. 用户可以在 Windows 桌面环境中安装并打开应用。
2. 用户可以新增一条口语化工作记录，并在本地保存。
3. 用户可以配置自定义 AI API，并完成连接测试。
4. 用户可以将单条口语记录优化为周报可用任务条目。
5. 用户可以编辑 AI 优化后的内容并保存。
6. 用户可以按日、周、月、年查看对应任务记录。
7. 用户可以搜索、筛选、删除任务记录。
8. 用户可以生成某一周的周报文本并复制。
9. AI API 调用失败时，原始记录不丢失，界面提示清晰。
10. 关闭应用后重新打开，历史记录仍然存在。

## 15. 风险与待确认问题

### 15.1 主要风险

1. 不同 AI 服务的 API 兼容程度不同，需要优先支持 OpenAI-Compatible 接口。
2. 用户口语记录可能信息过少，AI 输出质量依赖输入质量。
3. API Key 本地存储涉及安全要求，需要使用系统安全存储或加密方案。
4. 周报模板格式如果过早复杂化，会影响 MVP 交付速度。

### 15.2 待确认问题

1. 是否只需要单机本地使用，还是未来需要多设备同步？
2. 周报输出是否需要固定模板，例如“本周完成 / 下周计划 / 风险问题”？
3. 是否需要 Word 导出作为 V1.0 必备功能？
4. 是否需要项目维度统计，例如每个项目本周完成多少条任务？
5. AI API 是否默认按 OpenAI-Compatible 格式设计？

## 16. MVP 建议范围

第一版建议只做以下核心功能：

1. Windows 桌面应用。
2. 快速记录口语内容。
3. 自定义 AI API 配置和测试。
4. 单条任务 AI 优化。
5. 本地 SQLite 保存。
6. 今日、本周、本月、本年视图。
7. 周报文本生成和复制。

该范围可以最短路径验证产品价值，并为后续模板、导出、统计、同步等能力留出扩展空间。
